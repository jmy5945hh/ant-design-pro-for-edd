# BUG-4 修复评分标准

**编号**：BUG-4 | **难度**：⭐⭐⭐ | **满分**：10 分

---

## 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 定位准确度 | 25% | 是否定位到 `setTimeout` 延时参数 |
| 修复正确性 | 30% | 修复后图表更新频率是否合理 |
| 根因理解 | 25% | 能否解释从「页面卡顿」到「定时器配置错误」的因果链 |
| 代码审查 | 20% | 是否能发现代码中可优化的其他问题 |

---

## 评分细则

### 0-2 分：未发现或修复无效

| 分数 | 表现 |
|------|------|
| 0 | 未提交修复，或修复内容与本题无关 |
| 1 | 尝试修改了图表样式（如高度、颜色），或修改了 `getActiveData()` 的数据生成逻辑但未触及定时器频率问题 |
| 2 | 发现是定时器触发太频繁，但改为删除定时器或只执行一次（`useEffect` 中不递归调用 `loopData`），导致图表完全停止更新——修复过头了 |

### 3-5 分：找到了问题但修复不完整

| 分数 | 表现 |
|------|------|
| 3 | 正确将 `setTimeout(loopData, 100)` 的延时改为一个较大的值（如 `500` 或 `1000`），虽然频率降低了但不是最佳值 |
| 4 | 正确将延时改回 `2000`，但无法解释为什么这个值合理，或者无法解释 `100ms` 为什么会导致卡顿 |
| 5 | 修复正确 + 能解释因果链：`100ms 定时器` → `每 100ms 调用 setActiveData` → `触发 state 更新` → `React 重渲染` → `页面闪烁 + CPU 升高` |

### 6-8 分：正确修复 + 深入理解

| 分数 | 表现 |
|------|------|
| 6 | 在 5 分基础上，能说明如何通过 Chrome DevTools Performance 面板定位此类问题（找到密集的 setTimeout 调用 + React 重渲染火焰图） |
| 7 | 在 6 分基础上，注意到 `getActiveData()` 使用 `Math.random()` 每次生成完全随机的数据——即使间隔正常，数据也缺乏连续性。提出改进方向（如基于上一轮数据做增量更新，或使用动画过渡平滑数据变化） |
| 8 | 在 7 分基础上，审查了定时器的 cleanup 逻辑——确认 `useEffect` 返回的清理函数中正确调用了 `clearTimeout(timerRef.current)`，以及 `timerRef` 的使用是合理的。能评价这是一个正确的 cleanup 模式 |

### 9-10 分：卓越修复

| 分数 | 表现 |
|------|------|
| 9 | 修复正确 + 全面分析了组件性能：定时器频率 + 随机数据策略 + `useMemo` 的合理使用 + cleanup 正确性。提出了 2 个以上可落地的优化建议 |
| 10 | 在 9 分基础上，将优化建议落实为代码改动（如使用 `useCallback` 包裹 `loopData`、用 `useRef` 存储上一轮数据实现增量更新），或编写了性能测试验证优化效果 |

---

## 验收检查项

修复完成后，评委将验证以下内容：

- [ ] 图表更新频率为每 1-3 秒一次（非 100ms）
- [ ] 图表曲线不再剧烈跳动，数值可阅读
- [ ] 页面滚动流畅，无明显卡顿
- [ ] Performance 面板录制显示 setTimeout 回调间隔合理
- [ ] 定时器 cleanup 逻辑正常（组件卸载后不再触发更新，不会出现内存泄漏警告）
- [ ] 修复未引入 TypeScript 编译错误或 Lint 告警
