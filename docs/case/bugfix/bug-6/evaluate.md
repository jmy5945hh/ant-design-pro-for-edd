## 答辩参考问题（评委用）

以下问题可用于答辩环节考察选手对 Bug 的理解深度：

1. **为什么缓慢拖拽不容易触发，而快速拖拽容易触发？**
   - 参考答案：缓慢拖拽时，每次 resize 事件之间有足够的时间让 React 完成 state 更新和重渲染，下一个 rAF 回调执行时闭包中的 `initConfig` 已经是更新后的值（因为组件重渲染了，`resizeRef.current` 指向了新闭包）。快速拖拽时，多个 rAF 回调在 React 完成重渲染之前排队执行，它们都捕获了同一个过时的 `initConfig`。

2. **`resizeRef.current = resize` 这行代码的意义是什么？为什么不能直接在 `useLayoutEffect` 中用 `resize` 而不通过 ref？**
   - 参考答案：如果直接在 `useLayoutEffect` 的依赖数组中不传任何依赖（`[]`），闭包中的 `resize` 永远是首次渲染的版本，其中不仅 `initConfig` 过期，`mode` 的计算逻辑也可能过期（如果后续修改了阈值）。通过 `resizeRef.current = resize` 在每次渲染时更新 ref，保证事件处理器调用的是最新的 `resize` 函数定义。但这只解决了「函数定义」的新鲜度，没有解决「闭包值」的新鲜度。

3. **如果用 `useReducer` 替代 `useState` 来管理 `initConfig`，能避免这个问题吗？为什么？**
   - 参考答案：能。`useReducer` 的 `dispatch` 函数引用是稳定的（不随重渲染变化），且 reducer 函数接收的是 React 保证的最新 state，不存在闭包捕获旧值的问题。这是从根本上消除此类 Bug 的一种架构选择。
