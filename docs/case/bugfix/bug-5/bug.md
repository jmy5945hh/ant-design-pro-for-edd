# BUG-5：基础表单缺少防重复提交机制

**编号**：BUG-5

**页面路径**：表单 → 基础表单（`/form/basic-form`）

**发现版本**：竞赛分支

**严重程度**：严重（可能导致重复数据，在支付/订单场景下是一级事故）

**复现率**：约 80%（取决于点击速度）

---

## 测试环境

- 浏览器：Chrome 最新版
- 设备：桌面端（1920×1080）

---

## 测试步骤

1. 进入「表单」→「基础表单」页面
2. 依次填写各表单项：
   - 标题：输入 `测试项目`
   - 起止日期：选择任意日期范围
   - 目标描述：输入 `完成前端优化`
   - 衡量标准：输入 `性能提升 20%`
   - 客户（选填）：留空
   - 邀评人（选填）：留空
   - 权重（选填）：留空
   - 目标公开：选择「公开」
3. 打开 Chrome DevTools → Network 面板，切换到「All」或「Fetch/XHR」
4. **快速连续点击两次**「提交」按钮（模拟用户手抖双击或网络延迟时的重复点击）
5. 观察：
   - 提交按钮的状态（是否有 loading 动画、是否变为禁用状态）
   - Network 面板中的请求数量
   - 提示消息出现的次数

---

## 预期结果

1. 第一次点击提交后，按钮立即显示 loading 状态并**变为不可点击**
2. Network 面板中只有 **1 个** POST 请求发出
3. 请求成功后，提示「提交成功」**1 次**
4. 提交成功后按钮恢复可点击状态

---

## 实际结果

1. ❌ 提交按钮**没有 loading 动画**，点击后仍然可以再次点击
2. ❌ Network 面板中出现 **2 个 POST 请求**（快速双击时），请求 URL 和参数完全相同
3. ❌ 页面弹出「提交成功」提示 **2 次**
4. ❌ 如果后端未做幂等校验，数据库中会产生两条重复数据

**补充**：即使在较慢的网络条件下（Chrome DevTools → Network → Throttling → Slow 3G），双击仍然能成功发送两次请求，因为在请求 pending 期间按钮并未禁用。

---

## 附加信息

- 该页面使用 `@ant-design/pro-components` 的 `ProForm` 组件
- 提交逻辑使用了 `@tanstack/react-query` 的 `useMutation`
- 提示：关注 `useMutation` 返回的 `isPending` 状态，以及 `ProForm` 的 `submitter` 自定义能力
- 这是一个典型的**缺少防重复提交机制**的问题——前端应承担第一道防线
