# BUG-2 修复评分标准

**编号**：BUG-2 | **难度**：⭐⭐⭐ | **满分**：10 分

---

## 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 定位准确度 | 25% | 是否找到 `onSuccess` 回调中缺失的刷新逻辑 |
| 修复正确性 | 35% | 修复后删除操作是否触发表格刷新 |
| 根因理解 | 25% | 能否解释 ProTable 数据刷新机制与 React Query 的关系 |
| 工程素养 | 15% | 是否考虑了边界情况和其他刷新入口 |

---

## 评分细则

### 0-2 分：未发现或修复无效

| 分数 | 表现 |
|------|------|
| 0 | 未提交修复，或修复内容与本题无关 |
| 1 | 尝试修改了提示文案（如把 "will refresh soon" 改成了别的内容），未触及数据刷新的根本问题 |
| 2 | 找到了删除相关代码，但修改了错误的方向（如修改 `removeRule` 接口调用、或修改 `setSelectedRows([])` 逻辑） |

### 3-5 分：接近正确但理解不够

| 分数 | 表现 |
|------|------|
| 3 | 意识到应该调用某个刷新方法，但调用错误——如只添加了 `actionRef.current?.reload()` 而没注意到 `reloadAndRest()` 会重置分页 |
| 4 | 正确添加了 `actionRef.current?.reloadAndRest?.()`，但放在 `invalidateQueries` 之后或位置不当，且无法解释为什么这一行是必要的 |
| 5 | 修复正确，能说出「ProTable 需要通过 actionRef 来触发表格刷新」，但无法解释为什么已有的 `queryClient.invalidateQueries` 不生效 |

### 6-8 分：正确修复 + 深入理解

| 分数 | 表现 |
|------|------|
| 6 | 修复正确 + 能清楚解释：ProTable 的 `request` 属性传入的 `rule` 是原始请求函数，不是 React Query 的 queryFn，表格内部独立管理数据状态，所以 `invalidateQueries` 对表格不生效 |
| 7 | 在 6 分基础上，能解释 `reload()` 与 `reloadAndRest()` 的区别（后者会重置 pagination 到第一页），并说明此处选择 `reloadAndRest` 的原因 |
| 8 | 在 7 分基础上，分析出 `invalidateQueries` 在此场景虽不驱动表格刷新，但仍应保留的理由（可能有其他组件消费同一 `queryKey`），或指出了 `setSelectedRows([])` 在第 39 行的作用 |

### 9-10 分：卓越修复

| 分数 | 表现 |
|------|------|
| 9 | 修复正确 + 全面分析了 `onSuccess` 中三个操作（清空选中行 → 刷新表格 → 失效缓存）各自的职责和必要性，并验证了 UpdateForm 中 `onOk={actionRef.current?.reload}` 也正确使用了 actionRef |
| 10 | 在 9 分基础上，提出防止此类 Bug 回归的方案——如在 `useMutation` 封装层统一处理 ProTable 刷新，或编写测试用例验证删除后 `reloadAndRest` 被调用 |

## 验收检查项

修复完成后，评委将验证以下内容：

- [ ] 勾选 2-3 条数据 → 点击批量删除 → 表格自动刷新，被删除的数据消失
- [ ] 如果被删除的数据在当前页最后几条，刷新后分页是否正确回退
- [ ] 删除成功后提示消息正常显示
- [ ] 行内单条删除（通过 UpdateForm）功能不受影响
- [ ] Network 面板中删除后自动发出了新的列表数据请求
- [ ] 修复未引入 TypeScript 编译错误或 Lint 告警
